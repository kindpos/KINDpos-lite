# KINDpos/lite vs Toast POS — Feature Comparison

> Internal analysis — not marketing copy. Based on KINDpos/lite codebase audit (2026-04-04) and public Toast POS documentation/behavior.

---

## Comparison Table

| Feature | KINDpos | Toast | Operator Impact |
|---|---|---|---|
| **Item Selection UX** | 3-level hexagonal SVG bloom menu (HexNav). Tap category → subcategories bloom outward → tap subcat → items bloom. Tap any locked (filled) hex to go back one level. Spatial layout is deterministic — items always appear in the same relative position. No scrolling, no pagination. (`hex-nav.js`) | Flat grid of colored tiles. Categories across the top as tabs, items below as a scrollable grid. Search bar for large menus. Standard pagination for overflow. | KINDpos builds muscle memory through spatial consistency. Servers stop reading labels after a week — they know where the hex is. Toast requires scanning a grid every time, especially after menu changes shuffle tile positions. For a 80+ item menu, KINDpos eliminates scrolling entirely. |
| **Modifier Workflow** | Two-tab system: Items tab (HexNav shows menu items) and Modifiers tab (HexNav shows modifier categories). Select a prefix first (Add/No/On Side/Extra/Sub — color-coded buttons), then tap modifier from HexNav. Modifier is recorded as `"{Prefix} {Modifier}"` (e.g., "No Onion", "Extra Cheese", "Sub Tofu"). Same HexNav bloom navigation for modifiers as for items. (`order-entry.js:132-139, 420-428`) | Modifier groups appear as a forced pop-up after adding certain items (configured per item). User taps modifiers from a flat grid. Prefix logic is implicit — separate "No" and "Extra" buttons per modifier, or a toggle. No universal prefix system. | KINDpos decouples prefix from modifier — any prefix works with any modifier. Toast hardcodes modifier options per item. If a customer says "no bacon on anything," KINDpos handles it in 3 taps (select items, set prefix to No, tap Bacon). Toast requires editing each item individually. |
| **Multi-Select + Batch Modifier (Universal Modifier Pattern)** | Tap a collapsed group card to select all instances of that item. Tap individual instance cards to toggle selection. Multiple items can be selected simultaneously (mint highlight = selected). Switch to Modifiers tab → set prefix → tap modifier from HexNav → modifier is applied to ALL selected items in one action. (`order-entry.js:421-428, 511-513, 564-565`) | No multi-select for modifier application. Modifiers are applied per-item during or after item entry. Batch modifier application does not exist. | **Major differentiator.** A table of 6 orders a burger — select all 6, switch to modifiers, tap "No Pickle" once. Done. On Toast, that's 6 separate modifier edits. For high-volume counter service or catering orders, this cuts modifier entry time by 50-80%. |
| **Prefix System (Add/No/On Side/Extra/Sub)** | 5 universal prefix buttons, each color-coded: Add (green), No (red), On Side (gold), Extra (cyan), Sub (lavender). Active prefix persists until changed. Applies to the next modifier selection from HexNav. Prefix + modifier name concatenated as the mod label. (`order-entry.js:132-139`) | No universal prefix concept. Each modifier item in Toast is pre-configured as its own entry (e.g., "No Onion" is a separate modifier from "Extra Onion"). Modifier groups are defined per menu item in the back office. | KINDpos prefix system means 5 prefixes × N modifiers = 5N combinations with zero back-office setup. Toast requires pre-building every combination as a separate modifier entry. Adding a new modifier to Toast means creating up to 5 variants. KINDpos: add it once, all prefixes work automatically. |
| **Ticket Item Selection Model** | Items in the ticket panel are grouped by name. Collapsed groups show quantity + total price. Tap a group to expand and select all instances. Tap individual instances to toggle selection (mint highlight). Selected items are the target for modifiers, void, and delete. Deselect by tapping again; group auto-collapses when nothing is selected. (`order-entry.js:460-573`) | Flat list of line items. Tap one item to edit it. No group-by-name collapsing, no multi-select. Swipe-to-delete on some versions. | KINDpos ticket panel handles 30+ items without scrolling fatigue. Grouped view means "3× Brisket" is one row until you need to differentiate them. Toast shows every line item individually, requiring more scrolling on large orders. |
| **Tip Adjustment** | Dedicated scene with filterable table (All / Unadjusted toggle). Tap cyan tip cell → numpad appears, rest of UI dims to 30% opacity. After entering tip, cursor **auto-advances to next unadjusted check**. Batch-zero button sets all remaining unadjusted tips to $0 in one action. Retry logic (3 attempts, 1s/2s backoff) with red outline on failed rows. (`tip-adjustment.js:270-288, 502-547`) | Tip adjustment is per-check: navigate to closed check, tap "Adjust Tip", enter amount, save, navigate back to list, find next check, repeat. No auto-advance. No batch-zero. No visual indicator of which tips failed to sync. | A server closing out 30 checks on Toast: 30 × (find check + tap + enter + save + navigate back) = ~5 minutes. On KINDpos: tap first cell, enter amount, it jumps to next — enter, jump, enter, jump — batch-zero the rest. Under 60 seconds. Auto-advance alone saves 2+ taps per check. |
| **Tip Adjustment Checkout Gate** | When attempting checkout with unadjusted tips, an interrupt offers "Set to $0" or "Go Back" — not just a warning. The gate provides a resolution action within the dialog itself. (`tip-adjustment.js:549-596`) | Toast warns "You have unadjusted tips" with a Continue/Cancel dialog. No option to resolve within the dialog — user must cancel, go back, and manually zero each tip. | KINDpos eliminates a round-trip. The gate says "here's the problem, here's the fix, one tap." Toast makes you dismiss the warning, navigate to each check, zero it, then try again. This pattern repeats across KINDpos — interrupts guide toward resolution, not just inform. |
| **Server Checkout Flow** | 6-card accordion with expand/collapse. Cards color-code status: red = blocker (open checks), yellow = warning (unadjusted tips), mint = ready. Blocked cards include **jump navigation buttons** that take the user directly to the scene where they can fix the issue (e.g., "→ Tip Adjustment (3 remaining)"). FINALIZE button disabled with dashed outline when blocked. Manager PIN required to finalize. (`server-checkout.js:41-43, 800-815, 1094-1154`) | Server checkout is a single summary screen. If blocked, Toast shows a red banner saying "Close open checks before clocking out." No navigation shortcut to fix the issue — user must manually navigate to the checks screen, find the open checks, close each one, then return to checkout. | KINDpos treats blockers as solvable problems, not stop signs. A server sees "3 unadjusted tips" with an arrow button → taps it → lands in tip adjustment → fixes tips → pops back to checkout. On Toast, that's: dismiss warning → navigate to Checks → filter to open → close each → navigate back to Checkout. The jump buttons save 4-6 navigation steps per blocker. |
| **Day Close** | Two-phase process: (1) SUBMIT BATCH settles card transactions via Win98-style dialog with 26-segment progress bar and cycling status messages, (2) CLOSE DAY finalizes with manager PIN approval. Separated so manager can verify batch totals before committing to day close. 6-card accordion shows Revenue Summary, Payment Breakdown, Category Sales, Check Stats, Daypart Summary, Tips & Gratuity. (`close-day.js:642-759, 962-1018`) | Single "End of Day" button. Batch settlement and day close are combined. Summary shows as a printable report after close. No intermediate verification step. No animated progress for batch settlement — just a spinner. | Two-phase close reduces errors. Manager can catch a missing batch or incorrect total before the day closes. The Win98 progress dialog shows real batch processing steps — not an indeterminate spinner. Toast's combined close means any batch error cancels the entire operation, requiring a restart. |
| **Hardware Setup** | Manager-only Settings scene with multi-step wizard: choose device type → scan network (SSE-streamed, devices appear as found) OR enter IP manually → review results → confirm and save. MAC-as-identity means devices survive IP changes. Auto-detects Dejavoo card readers via SPIn probe. Test print sends branded ESC/POS receipt. Hot-reload after config change — no restart. (`settings.js`, `hardware.py`) | Hardware setup is done through Toast's web-based back office (not on the terminal). Printers and card readers are pre-configured by Toast installation team. Network scanning is not available on the terminal. Adding a device requires a support call or back-office login. | KINDpos puts hardware management on the terminal itself. A manager can add a new printer during service without leaving the POS, without calling support, without opening a laptop. SSE streaming means they see devices appear in real-time during scan. Toast's model requires either a support call or back-office access — neither available during a rush. |
| **Diagnostic / Audit Trail** | Hash-chained event ledger (SHA-256, append-only SQLite). Every business event (order, payment, void, print) is recorded with prev_hash → hash linkage. Tampering with any historical event breaks the chain. Separate Entomology diagnostic system: hash-chained diagnostic events, adaptive heartbeat (60s active / 15min idle), 3-layer HTML report (health summary, pattern analysis, event timeline), reverse correlation for root-cause linking. (`event_ledger.py`, `diagnostic_collector.py`, `entomology_report.py`) | Toast logs transactions to its cloud backend. Audit trail is available through the back office as filterable reports. No hash chaining — integrity depends on server-side access controls. No on-device diagnostic system. No heartbeat monitoring. Diagnostic info requires contacting Toast support. | KINDpos audit trail is cryptographically tamper-evident at the device level — not dependent on cloud trust. The Entomology report gives the operator (or a tech) a full diagnostic picture without calling support: "Layer 1 says printer went offline 3 times yesterday, Layer 2 shows it happens during the lunch rush, Layer 3 has the exact timestamps." Toast requires a support ticket for this level of detail. |
| **Navigation Architecture** | Three-tier system: Scenes (full-screen, stack-based push/pop/replace with canExit guards), Overlays (semi-modal, z:100, pauses scene), Interrupts (Promise-based blocking modal, z:200, one at a time). 200ms debounce prevents double-tap issues. Scene caching for instant resume. Prefetch pre-builds upcoming scenes. Idle timeout per scene. 11 diagnostic events emitted for every nav action. (`scene-manager.js`) | Standard page-based navigation. Modals for confirmations. No layered navigation system. No exit guards (user can navigate away from unsaved orders). No prefetch. No navigation-level diagnostics. | KINDpos exit guards prevent data loss — canExit() blocks navigation if there's unsaved work. The three-tier separation means overlays and interrupts are architecturally distinct, not just CSS tricks. Toast's flat navigation means a misplaced tap can lose an in-progress order. The 200ms debounce alone prevents the "double-tap sent two orders" bug that plagues touch POS systems. |
| **Button / Touch Feedback** | Every button uses chamfered clip-path polygon (8px corners, no border-radius anywhere). 3D bevel edges (light top/left, dark bottom/right). On press: bevel inverts, shadow disappears, element translates (3px, 4px) — button physically pushes in. 50ms transition. Color-coded bevel lookup table per fill color. Sunken style for inputs (inverted bevel). (`tokens.js`, `base.css`) | Material Design-inspired flat buttons with ripple animation on tap. Slight color darkening on press. No 3D depth simulation. Standard border-radius rounding. | KINDpos press feedback is unambiguous — the button visually depresses into the surface. On a greasy touchscreen in a loud kitchen, "did I tap that?" is a real question. KINDpos answers it with a physical metaphor: pushed in = registered. Toast's ripple can be missed in bright light or peripheral vision. The chamfered design also creates a distinctive brand identity that doesn't look like "another iPad app." |
| **Pricing Transparency (Dual-Price)** | Receipt-review scene shows card total AND cash price side by side with explicit savings amount ("save $X.XX with cash"). Cash discount rate is configurable (default 4%). Both prices are calculated from the same subtotal with clear derivation. Customer sees both options before choosing payment method. (`receipt-review.js:255-258`, `order-entry.js:21-22`) | Single price displayed. Cash discount programs exist via third-party integrations (e.g., DualPricing by CardX) but are not native to Toast. When available, the discount is shown as a line item adjustment, not a side-by-side comparison. | KINDpos makes the cash discount a first-class UX element. The customer sees "Card: $42.80 / Cash: $41.09 / Save $1.71" as a visual choice, not a receipt footnote. This drives cash adoption, which reduces processing fees. Toast's approach (when available via add-on) buries the discount in receipt math. |
| **Void Workflow** | Two behaviors based on item state: unsent items silently delete (no gate, no audit — they were never sent to kitchen). Sent items trigger a two-step interrupt chain: (1) manager PIN entry via numpad, (2) structured reason selection (Customer Changed Mind / Wrong Item / Quality Issue / Manager Request / Other). Creates full audit trail. VOID button dynamically relabels to DELETE for unsent items. (`order-entry.js:618-639, 974-975`) | Single void flow regardless of item state. Manager approval required for all voids (configurable). Reason selection is a text field or simple dropdown. No distinction between "never sent" and "already in kitchen." | KINDpos recognizes that removing an item you haven't sent yet is not a void — it's just editing your order. This eliminates unnecessary manager interruptions for innocent corrections. Toast requires manager approval even to remove an item added 2 seconds ago. For KINDpos, the two-step chain (PIN → reason) for real voids creates a clean audit trail without burdening the operator for non-events. |
| **Save / Recall (Tab Management)** | SAVE button opens QWERTY keyboard for optional tab name (max 20 chars) → saves to localStorage. RECALL button opens 3-column overlay grid of saved tabs → tap to get multi-option interrupt (RECALL / DELETE) → if current ticket exists, second interrupt asks to confirm clear. Three-layer interaction: overlay → interrupt → conditional interrupt. (`order-entry.js:793-968`) | "Hold" button saves an order to the cloud. Recall via a held orders list. No naming — identified by table number or check number. No local storage — requires network connectivity. | KINDpos tabs work offline (localStorage). Named tabs ("Mike's lunch", "Catering order") are easier to find than check #4072. The multi-option recall (RECALL/DELETE with overwrite guard) prevents accidents. Toast's cloud-dependent hold fails silently when WiFi drops during a rush. |
| **Payment Auto-Return** | After payment completes, a 4000ms progress bar counts down then auto-navigates back to order-entry via `replace()` (no back-nav possible). Tap anywhere to dismiss early. (`payment.js:409, 435-437, 452`) | After payment, Toast shows a confirmation screen. Requires explicit "New Order" tap to proceed. No auto-return. | Auto-return shaves 2-3 seconds per transaction. Over a 200-transaction day, that's 7-10 minutes recovered. The `replace()` navigation prevents the accidental back-tap that would reopen a completed payment — a common Toast complaint. |
| **Role-Based Scene Visibility** | Reporting scene physically swaps cards: server sees "Checkout," manager sees "Close Day." Settings scene is invisible to non-managers (silent gate — button does nothing, no error). Role determines which scenes exist, not which buttons are grayed out. (`login.js:129`, `reporting.js:139-162`) | Role permissions disable buttons (grayed out with lock icon) or show "Access Denied" toast notification. The UI structure is the same for all roles — restricted items are visible but inaccessible. | KINDpos shows each role only what they can use. A server never sees "Close Day" — they don't know it exists on their screen. This reduces confusion and eliminates the "why can't I tap this?" question. Toast's approach of showing locked buttons invites curiosity and support calls. |
| **Interrupt System (Modal Design)** | Promise-based interrupts at z-index 200 with 70% dim. Return structured values (not just boolean). Support multi-option responses ("Set to $0" / "Go Back"). Can chain sequentially (void-pin → void-reason). Only one active at a time. Enforced at navigation layer, not per-scene. (`scene-manager.js:273-316`) | Standard JavaScript confirm/alert patterns or custom modal components. Binary confirm/cancel. No chaining. No structured return values. Modals are per-page, not system-wide. | KINDpos interrupts are decision trees, not speed bumps. When the system blocks you, it offers paths forward — not just "OK" or "Cancel." The Promise-based architecture means scenes can `await` a multi-step approval flow in a single code path. Toast's binary modals can't express "here are 3 things you could do right now." |
